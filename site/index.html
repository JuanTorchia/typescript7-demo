TypeScript 7.0 native preview

The biggest TypeScript runtime shift is here.

TypeScript 7 moves the compiler to a native Go foundation, adds parallelism, and is designed to run side-by-side with TypeScript 6 while the ecosystem migrates. This lab turns the announcement into real checks you can run.

What's new in TypeScript 7

Why developers should care.

Native compiler foundation

The compiler has been ported from a JavaScript runtime model to Go, targeting native-code speed while preserving TypeScript 6 semantics.

Parallel type-checking

New worker-based checking can use multiple cores. The preview exposes controls like --checkers and single-threaded mode for tuning and debugging.

Project reference parallelism

Larger repos and monorepos can benefit from parallel project builds through builder controls, with memory trade-offs teams should measure.

Side-by-side migration

The preview ships as @typescript/native-preview with tsgo, while TypeScript 6 can run as tsc6.

Faster editor direction

The same native foundation powers the preview editor experience, so speedups should matter beyond CI and into daily feedback loops.

Real migration cleanup

TypeScript 7 hard-errors on several deprecated options. That is annoying, but useful: it tells teams exactly what must be cleaned before the stable release.

Stricter object-key typing

The project includes an example where the key type must be narrowed to string keys because Object.keys never returns symbols. 

export function typedKeys<T extends Record<string, unknown>>(
  value: T,
): Array<Extract<keyof T, string>> {
  return Object.keys(value).filter(
    (key): key is Extract<keyof T, string> => key in value,
  );
}

Declaration safety

The repo keeps a fixture that intentionally fails with isolatedDeclarations, proving the test suite catches public API annotations that need to be explicit.

Decision guide

What should you do with the result?

01

Both compilers pass

Good signal. Keep the workflow manual or scheduled weekly to watch the preview evolve without changing your production compiler.

02

TypeScript 7 fails on config

Treat it as migration work, not a runtime bug. Update deprecated options, project references, module resolution, or declaration settings in a branch.

03

TypeScript 7 is faster

Useful for planning. If type-checking blocks local dev or CI, this gives you evidence to follow the beta closely.

04

TypeScript 7 is slower or unsupported

Also useful. You now have a reproducible issue shape and can decide whether to wait, reduce config complexity, or report a preview limitation.

Security model

We do not need access to your code.

The safest version is a CI recipe, not a hosted remote-code execution service. You add the workflow to your own repository, run it manually, and keep the logs, artifacts, and permissions under your GitHub account.

  • No repository clone on our servers.
  • No private source code leaves your GitHub Actions runner.
  • No write permissions are required for the workflow itself.
  • You can delete the workflow at any time.

Generated workflow

This workflow compares your current type-check command with tsgo --noEmit. 

name: TypeScript 7 Preview Check

on:
  workflow_dispatch:

permissions:
  contents: read

jobs:
  check:
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Install TypeScript 7 native preview
        run: npm install --no-save @typescript/native-preview@beta

      - name: Current project type-check
        run: npm run typecheck

      - name: TypeScript 7 native preview version
        run: npx tsgo --version

      - name: TypeScript 7 native preview check
        run: npx tsgo --noEmit
How to interpret results

Use it as early migration signal.

A passing run means the preview compiler can type-check your project under this command today. It does not mean TypeScript 7 is production-ready for your entire toolchain.

A failing run is still useful. It can reveal unsupported compiler options, stricter type behavior, project-reference edge cases, or preview limitations.
